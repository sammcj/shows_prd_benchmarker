Top Gaps & Why They Matter
5 highest-impact items
PRD-002 Critical
User's overlaid data must display on every surface
The plan implements user data in individual components (tiles, detail page, search) but lacks a cross-cutting contract guaranteeing it. Without this explicit principle, new surfaces — AI recommendation cards, Alchemy results, mentioned-shows strips — will render shows without user data overlays. This is the highest-severity gap because it's a foundational UX rule. A user who rated and tagged a show would see it appear as a "bare" catalog item in Alchemy results.
PRD-103 Important
Data sync has no transport mechanism
The plan includes per-field merge logic for conflict resolution, but describes no mechanism for cross-device sync — no Supabase Realtime subscriptions, no polling, no trigger. The merge logic exists in a vacuum. A user opening the app on a second device would find their collection empty with no path to retrieve it, immediately eroding trust in the product.
PRD-104 Important
No upgrade/migration strategy for existing data
The plan has an app_metadata table with data_model_version and numbered SQL migrations, but no migration runner or strategy to transform existing rows when the schema evolves. If a field is added or renamed, existing user data could be lost or produce errors after an update.
PRD-008 Important
Removal warning lacks progressive suppression
The plan has a confirmation dialog for show removal but omits the mechanism where, after several confirmations, the user can opt to stop seeing the warning. This maps to the hideStatusRemovalConfirmation and counter in the storage schema. Power users managing large libraries face unnecessary friction on every removal.
PRD-054 Important
Explore Similar: concept changes don't clear results
The plan covers this rule for Alchemy ("changing shows clears concepts/results") but does not state the equivalent for Explore Similar. If a user changes their concept selection, stale recommendations from the previous selection would remain visible, misrepresenting the AI's actual output for the new input.

Risk Assessment
What breaks first if we ship as-is

Most likely user-visible issue: Subtle inconsistency in how shows display across surfaces. A show the user has rated and tagged would correctly show its status badge on the Home page and in Search results, but appear as a "bare" catalogue item in an Alchemy recommendation row or a mentioned-shows strip in Ask. A QA reviewer would notice first that the Alchemy results card for a show already in the collection doesn't carry the user's rating badge.

Second likely catch: A user who opens the app on a second device and finds their collection empty because no sync subscription or poll exists. The merge logic is sound, but without a transport layer, it never fires. This would immediately erode trust, even if the data is safe server-side.

Gap pattern: The gaps are not randomly scattered. They cluster around two themes: (1) lifecycle and state management for ephemeral data — the plan doesn't consistently articulate that Alchemy results, Ask history, and mentioned-shows strips are session-only; and (2) cross-cutting UX polish rules from the supporting documents (detail_page_experience.md, storage-schema.md) that weren't consulted with the same rigour as the main PRD.

Remediation Path
Three targeted workstreams to reach 100%
1
Data Lifecycle Addendum
Add a section (or amendments to Phase 2) explicitly specifying which data is ephemeral, what events clear it, and how concept-selection state changes propagate. Covers PRD-054, PRD-097, PRD-098, PRD-099. This is a specification gap, not an architectural one — the code structure already supports it.
2
Cross-Cutting Display Contract
Elevate PRD-002 to an explicit principle with a utility or hook (e.g. useShowWithOverlay) that all show-rendering surfaces consume. This ensures every surface — tiles, search results, AI recommendations, mentioned-shows strips — consistently displays the user's My Data overlay.
3
Sync Transport & Migration Strategy
Genuine architectural decisions needed: Will Supabase Realtime subscriptions be used for sync? Will migrations run automatically on app load or via a separate script? These are not fit-and-finish items — they need planning before implementation begins. Covers PRD-103 and PRD-104.
